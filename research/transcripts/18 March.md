## The honest verdict

The normalised approach earns its keep when you query *across* exams — "find all exams where the subjective right eye SPH is stronger than -4.00", for example. That's a legitimate clinical reporting query. But it's not in scope for this system, which is a management/scheduling tool, not a research database.

For the actual use case here — render a form, save a form — normalisation adds:
- A `groupBy` + 8 `?->first()` calls on every page load
- 8 `updateOrCreate` calls on every save
- A translation layer between HTML field names and model structure

...with no corresponding benefit.

**Recommendation reversal: go flat columns on `exam_refraction`.**

The table will be wide (~60 columns), but that's an honest reflection of the domain — there genuinely are ~60 distinct measurements on this form. In the Blade template you get:

## Design Patterns — Where They Fit Naturally

### Already agreed

**Repository** — `PatientRepository`, `ExaminationRepository`, `AppointmentRepository`. Controllers ask for what they need, Eloquent stays out of them. The patient search screen (7 filter combinations, multiple sort options) is the clearest argument for this: that query logic has no business living in a controller.

**Factory** — `ExaminationFactory::create(ExamType $type, Patient $patient): Examination`. Right now it just creates a Spectacle exam and its four child rows atomically. When Red Eye and Contact Lens exams are added later, the factory absorbs the branching — the controller never changes.

**Observer** — `AppointmentObserver::created()` dispatches `SendAppointmentConfirmationJob`. `ExaminationObserver::created()` dispatches whatever post-save jobs are needed (PDF generation, recall scheduling). The queued job architecture means the observer stays thin — it just fires and forgets.

---

### Where Strategy fits naturally — two real cases

**1. Notification channel** — the "Update & Notify" button on appointments is the obvious one. *How* you notify a patient (SMS, email, a printed letter for patients without contact details) is a decision that changes per patient and potentially per diary. A `NotificationStrategy` interface with concrete `SmsNotification`, `EmailNotification`, `LetterNotification` implementations means the appointment service calls `$strategy->notify($appointment)` and never cares which channel is used. This is a genuinely natural fit because the reference doc already identifies the "Update & Notify" action as distinct from a plain update.

**2. Recall scheduling** — when an exam is signed off, `retest_after` determines when the patient should be recalled (1 year, 2 years, etc.). Different patient types may also have different recall rules imposed by NHS guidelines (e.g. diabetic patients on shorter recall cycles). A `RecallStrategy` that takes a patient and exam and returns a recall date is cleaner than a chain of if/else inside a job.

---

### Where Value Object fits naturally — and why it's worth adding

This one isn't in your original list but it earns its place. The Rx measurement block (SPH, CYL, Axis, Prism, Direction) appears six times across the refraction form. The "Transpose Rx" button applies a specific optical calculation to that block. That calculation is domain logic that has nothing to do with persistence — it belongs in an `RxMeasurement` value object that knows how to transpose itself. This also gives you a natural home for validation rules (Axis must be 0–180, SPH is typically ±30, etc.) that don't belong on the Eloquent model.

---

### Where Singleton does *not* fit naturally

Singleton is the honest answer to skip. Modern Laravel's service container already manages singleton bindings via `app()->singleton()` — adding a classic PHP Singleton on top would be redundant and would actually undermine testability (static state is hard to reset between tests). If you need a single shared instance of something (e.g. a configuration loader), just register it as a singleton in a Service Provider and inject it. The pattern is already handled by the framework.

---

### Pattern map summary

| Pattern | Where it lives | Trigger |
|---|---|---|
| **Repository** | `PatientRepository`, `ExaminationRepository`, `AppointmentRepository` | All data access from controllers |
| **Factory** | `ExaminationFactory` | Creating a new exam + its child tab rows atomically |
| **Observer** | `AppointmentObserver`, `ExaminationObserver` | Dispatch queued jobs on create/save |
| **Strategy** | `NotificationStrategy` | "Update & Notify" — channel varies per patient |
| **Strategy** | `RecallStrategy` | Recall date calculation after exam sign-off |
| **Value Object** | `RxMeasurement` | Transpose Rx logic, optical field validation |
| ~~Singleton~~ | ~~n/a~~ | Handled by Laravel's service container — don't add |

## 🤖 Claude

**Backed enums in `app/Enums/`, always.** Here's why the inline approach loses on this codebase specifically:

---

### The case against inline cast arrays

The moment you write `'status' => ['active', 'deceased', 'deleted']` on the model, you've created a string that has to be repeated verbatim in:
- The FormRequest validation rule
- The Repository filter query
- Any Blade dropdown that needs the options
- Any job that branches on the value

When `PatientType` appears on `patients.patient_type` *and* `exam_refraction.retest_patient_type`, you now have two models that both need to agree on the same set of strings. There's no enforcement that they do.

---

### What backed enums give you that matters here

**1. Single source of truth across models.** `PatientType::class` is cast on both `Patient` and `ExamRefraction`. One definition, impossible to drift.

**2. A `label()` method you'll use constantly.** Every dropdown in the UI needs a human-readable string. With a backed enum:

```php
enum PatientType: string
{
    case Private       = 'A';
    case Over60        = 'B';
    case NHS           = 'C';
    case FamilyHistory = 'D';
```

  ```php
  {
      $reqStart = $this->timeToMinutes($startTime);
      $reqEnd   = $reqStart + $lengthMinutes;
      $existing = Appointment::where('diary_id', $diaryId)
          ->whereDate('date', $date)
          ->whereNull('cancelled_at')
          ->when($excludeId, fn ($q) => $q->where('id', '!=', $excludeId))
          ->get(['id', 'start_time', 'length_minutes']);
      foreach ($existing as $appt) {
          $apptStart = $this->timeToMinutes($appt->start_time);
          $apptEnd   = $apptStart + $appt->length_minutes;
          if ($reqStart < $apptEnd && $reqEnd > $apptStart) {
              throw ValidationException::withMessages([
                  'start_time' => 'This time slot is already booked.',
              ]);
          }
      }
  }
  ```
