  Supporting lookup tables

  - practices — id, name, address, etc.
  - doctors — id, name, (possibly practice_id)
  - pcts — id, name (the long list of hospitals/trusts, seeded)

  ▎ Decision point: Should dropped_reason, domiciliary_reason, and how_heard be 
  stored as free-text strings (seeded lookup enforced only in the UI), or as 
  proper FK lookup tables? Lookup tables are more normalized but add a lot of 
  overhead for fields the user will rarely query by. I'd lean toward PHP enums 
  backed by varchar columns for these — clean, no extra tables, still type-safe.

  ▎ Decision point: Soft delete covers both "deceased" and "deleted" states — 
  should we differentiate them? We could add a status column (active, deceased, 
  deleted) rather than just deleted_at. This keeps soft-delete semantics for 
  deletion while allowing a deceased flag without conflating the two.

  For the medication checkboxes, two options:

  ▎ Decision point A — Medications as pivot table:
  ▎ medications (id, name) seeded with ~50 drugs + examination_medications 
  (examination_id, medication_id). Fully normalised, easy to query "all patients
   on Warfarin". Costs an extra join.

  ▎ Decision point B — Medications as JSON:
  ▎ Store as a medications JSON column (array of strings). Much simpler, no 
  extra tables. Querying by medication is possible but awkward in SQL. Fine if 
  you never need "find all patients on drug X" from the database directly.

  My recommendation: pivot table — you're building a medical system and "which
  patients are on Warfarin?" is a plausible query, especially with the
  observer/job pattern. It also makes the medication list extensible.

  ---
  Design Patterns — Where They Fit Naturally

  Already agreed

  Repository — PatientRepository, ExaminationRepository, AppointmentRepository.
  Controllers ask for what they need, Eloquent stays out of them. The patient
  search screen (7 filter combinations, multiple sort options) is the clearest
  argument for this: that query logic has no business living in a controller.

  Factory — ExaminationFactory::create(ExamType $type, Patient $patient): 
  Examination. Right now it just creates a Spectacle exam and its four child
  rows atomically. When Red Eye and Contact Lens exams are added later, the
  factory absorbs the branching — the controller never changes.

  Observer — AppointmentObserver::created() dispatches
  SendAppointmentConfirmationJob. ExaminationObserver::created() dispatches
  whatever post-save jobs are needed (PDF generation, recall scheduling). The
  queued job architecture means the observer stays thin — it just fires and
  forgets.

  ---
  Where Strategy fits naturally — two real cases

  1. Notification channel — the "Update & Notify" button on appointments is the
  obvious one. How you notify a patient (SMS, email, a printed letter for
  patients without contact details) is a decision that changes per patient and
  potentially per diary. A NotificationStrategy interface with concrete
  SmsNotification, EmailNotification, LetterNotification implementations means
  the appointment service calls $strategy->notify($appointment) and never cares
  which channel is used. This is a genuinely natural fit because the reference
  doc already identifies the "Update & Notify" action as distinct from a plain
  update.

  2. Recall scheduling — when an exam is signed off, retest_after determines
  when the patient should be recalled (1 year, 2 years, etc.). Different patient
   types may also have different recall rules imposed by NHS guidelines (e.g.
  diabetic patients on shorter recall cycles). A RecallStrategy that takes a
  patient and exam and returns a recall date is cleaner than a chain of if/else
  inside a job.

  ---
  Where Value Object fits naturally — and why it's worth adding

  This one isn't in your original list but it earns its place. The Rx
  measurement block (SPH, CYL, Axis, Prism, Direction) appears six times across
  the refraction form. The "Transpose Rx" button applies a specific optical
  calculation to that block. That calculation is domain logic that has nothing
  to do with persistence — it belongs in an RxMeasurement value object that
  knows how to transpose itself. This also gives you a natural home for
  validation rules (Axis must be 0–180, SPH is typically ±30, etc.) that don't
  belong on the Eloquent model.

  ---
  Where Singleton does not fit naturally

  Singleton is the honest answer to skip. Modern Laravel's service container
  already manages singleton bindings via app()->singleton() — adding a classic
  PHP Singleton on top would be redundant and would actually undermine
  testability (static state is hard to reset between tests). If you need a
  single shared instance of something (e.g. a configuration loader), just
  register it as a singleton in a Service Provider and inject it. The pattern is
   already handled by the framework.

● Backed enums in app/Enums/, always. Here's why the inline approach loses on
  this codebase specifically:

  ---
  The case against inline cast arrays

  The moment you write 'status' => ['active', 'deceased', 'deleted'] on the
  model, you've created a string that has to be repeated verbatim in:
  - The FormRequest validation rule
  - The Repository filter query
  - Any Blade dropdown that needs the options
  - Any job that branches on the value

  When PatientType appears on patients.patient_type and
  exam_refraction.retest_patient_type, you now have two models that both need to
   agree on the same set of strings. There's no enforcement that they do.

  ---
  What backed enums give you that matters here

  1. Single source of truth across models. PatientType::class is cast on both
  Patient and ExamRefraction. One definition, impossible to drift.

  2. A label() method you'll use constantly. Every dropdown in the UI needs a
  human-readable string. With a backed enum:

  enum PatientType: string
  {
      case Private       = 'A';
      case Over60        = 'B';
      case NHS           = 'C';
      case FamilyHistory = 'D';
      case Child         = 'E';
      case Glaucoma      = 'F';

      public function label(): string
      {
          return match($this) {
              self::Private       => 'A — Private Patient',
              self::Over60        => 'B — Over 60',
              self::NHS           => 'C — NHS',
              self::FamilyHistory => 'D — Family History of Glaucoma',
              self::Child         => 'E — Child (under 16)',
              self::Glaucoma      => 'F — Glaucoma',
          };
      }
  }

  Blade dropdown becomes PatientType::cases() — no controller involvement, no
  hardcoded arrays in views.

  3. Native Laravel integration is already there. $casts accepts the enum class
  directly (Laravel 9+), and FormRequest validation uses
  Rule::enum(PatientType::class) — no custom logic needed.

  4. Type hints in the Factory and Repository. The ExaminationFactory can accept
   ExamType $type rather than string $type. PHP enforces at the call site that
  you're passing something valid.
